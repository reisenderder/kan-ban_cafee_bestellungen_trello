# Work Plans проекта

## Назначение раздела

Этот раздел хранит пошаговые планы реализации после Global Spec, Functional Map, Feature Specs, Technical Specs и User Stories.

Work Plans отвечают на вопрос: в каком порядке переносить утвержденное поведение в реализацию, какие зависимости учитывать и по каким критериям считать этап готовым.

## Правила раздела

1. Work Plan должен ссылаться на связанные Feature Specs, Technical Specs и User Stories.
2. Work Plan не добавляет новую бизнес-функцию.
3. Work Plan не должен менять смысл ролей, статусов, прав доступа или пользовательских сценариев.
4. Если для реализации нужен отдельный технический выбор, он фиксируется как предварительное решение или отдельный блокер.
5. Каждый Work Plan должен иметь проверяемые этапы и критерии готовности.
6. Новый Work Plan всегда создается в `active/`.
7. Work Plan остается в `active/` до завершения реализации и явного подтверждения пользователя после ручного тестирования.
8. Перед `git add`, `commit`, `push` или созданием pull request подтвержденный пользователем Work Plan обязательно перемещается в `completed/`.
9. Одновременно фактически выполненные работы добавляются в каждый затронутый журнал `../roadmap/`.
10. При перемещении нужно обновить статус плана, этот индекс и все ссылки на файл.
11. Завершенные Work Plans и связанные записи roadmap не удаляются и не перезаписываются новыми планами.

## Структура

* `active/` - планы, которые ожидают реализации, находятся в разработке или еще не подтверждены пользовательским тестированием.
* `completed/` - проверенные пользователем планы, по которым завершена разработка.
* `../roadmap/` - хронологическая история результатов завершенных планов по областям проекта.
