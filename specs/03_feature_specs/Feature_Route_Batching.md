# Feature Spec: Попутная доставка и группировка маршрута

> **Статус**: Черновик
> **Дата создания**: 2026-07-21
> **Версия**: 0.1
> **Источник**: `../01_global_spec/Global_Spec.md`, `../02_functional_map/Functional_Map.md`, `Feature_Order_CRM.md`, `Feature_Courier_Delivery.md`, `Feature_Order_Statuses.md`

---

## 1. Назначение

Эта функция описывает попутную доставку: когда менеджер может назначить одному курьеру несколько заказов, как система может предложить объединение по локациям, что видит курьер, кто решает фактический маршрут и какие ограничения нужны, чтобы группировка заказов не ломала базовый процесс доставки.

Попутная доставка нужна для экономии времени и ресурсов, когда несколько готовых заказов находятся в одном понятном направлении. На первом этапе один попутный маршрут может включать до трех заказов.

Система может помогать менеджеру и предлагать объединение заказов по локациям, геолокации и заметкам карты, которые клиент передал при оформлении заказа. При этом финальное решение остается за менеджером: система предлагает, менеджер решает.

---

## 2. Связь с Global Spec и Functional Map

Функция раскрывает следующие принципы:

* система поддерживает обычную доставку и возможность попутной доставки;
* система может предложить менеджеру объединение заказов по локациям;
* итоговые операционные решения остаются за менеджером и курьером;
* курьер видит только назначенные ему заказы;
* курьер не получает доступ к полной клиентской базе;
* после закрытия доставки доступ курьера к данным клиента закрывается;
* если у курьера есть следующие заказы, он не должен долго ждать недоступного клиента.

Функция опирается на закрытые решения:

* менеджер может назначить одному курьеру до трех заказов;
* заказы должны находиться в одном контуре маршрута;
* система может предложить подходящее объединение по локациям;
* менеджер решает, объединять ли предложенные заказы;
* система может предложить маршрут;
* курьер сам решает фактический порядок объезда;
* менеджер может отправить курьеру дополнительную заметку через канал связи;
* каждый заказ остается отдельной доставкой со своим QR, оплатой, статусом и историей.

## 3. Границы функции

### 3.1 Входит в функцию

* системное предложение объединения заказов по локациям;
* ручное подтверждение или отклонение объединения менеджером;
* назначение до трех заказов одному курьеру;
* отображение списка назначенных заказов курьеру;
* канал связи менеджера с курьером по маршруту;
* самостоятельный выбор фактического порядка объезда курьером;
* ограничения по данным клиента;
* отдельная фиксация доставки по каждому заказу;
* отдельное QR-подтверждение по каждому заказу;
* отдельная фиксация оплаты по каждому заказу;
* действия при недоступном клиенте в маршруте с несколькими заказами;
* история назначения маршрута.

### 3.2 Не входит в функцию

* полностью автоматическое назначение попутной доставки без менеджера;
* автоматическая оптимизация маршрутов;
* карта с обязательным расчетом лучшего маршрута;
* GPS-трекинг курьера в реальном времени;
* автоматическое объединение заказов системой без подтверждения менеджера;
* расчет стоимости доставки;
* техническая геокодировка адресов;
* продвинутые правила зон доставки;
* складской учет;
* управление графиком курьеров;
* платежная интеграция.

---

## 4. Участники

### 4.1 Менеджер

Менеджер решает, можно ли объединить несколько заказов в один маршрут. Он смотрит на адрес, ориентир, геолокацию, район, готовность заказов и текущую загрузку курьера.

Система может предложить менеджеру объединение, но менеджер несет ответственность за то, чтобы попутная доставка была реальной, а не просто формальным объединением заказов.

Если менеджер хочет дать курьеру дополнительную заметку по маршруту, он пишет ее в канале связи с курьером.

Менеджер сам решает, какие заказы комбинировать, каким заказам дать приоритет и какие заказы не объединять. Для этих решений автоматизация не нужна.

### 4.2 Курьер

Курьер получает список назначенных заказов. Он видит данные только по этим заказам и только на время доставки.

Система может предложить маршрут, но курьер сам определяет фактический порядок объезда внутри выданного маршрута.

### 4.3 Клиент

Клиент не должен видеть внутреннюю группировку маршрута. Для клиента не должно быть разницы, едет заказ напрямую или в попутной доставке.

### 4.4 Система

Система фиксирует, что несколько заказов назначены одному курьеру в рамках одного маршрута, и сохраняет историю действий.

Система может предложить менеджеру объединение заказов, если видит близкие или логически совместимые локации по данным карты, геолокации и клиентским заметкам.

---

## 5. Когда можно объединить заказы

На первом этапе объединение заказов возможно только по решению менеджера. Система может предложить объединение, но не должна назначать попутную доставку автоматически.

Менеджер может объединить заказы, если:

* в маршрут входит не больше трех заказов;
* заказы готовы к доставке;
* заказы находятся в одном направлении или районе;
* геолокации и адреса не создают явный конфликт маршрута;
* время приготовления заказов позволяет доставить их без сильного ухудшения качества;
* курьер реально может выполнить маршрут;
* у заказов нет проблем, которые делают попутную доставку рискованной.

Заказы из желтого или красного списка могут быть включены в попутную доставку, если выполнены все условия для принятия и доставки такого заказа. Для красного списка это означает, что требуемая онлайн-оплата уже произведена и подтверждена.

---

## 6. Когда не стоит объединять заказы

Менеджер не должен объединять заказы, если:

* один из заказов еще не готов;
* адрес или геолокация одного заказа сильно выбивается из маршрута;
* один из заказов требует срочного отдельного решения;
* один из клиентов находится в проблемной ситуации;
* один заказ требует полной онлайн-оплаты, которая еще не подтверждена;
* есть высокий риск, что первый клиент будет долго недоступен и задержит остальные доставки;
* менеджер считает, что состав заказа или качество блюда не подходит для попутной доставки.

Система не должна автоматически запрещать попутную доставку из-за типа блюда, качества блюда или приоритета. Менеджер сам решает, какие заказы комбинировать, какие заказы ставить в приоритет и какие не объединять.

---

## 7. Что видит курьер

Курьер видит только заказы, назначенные ему менеджером.

По каждому заказу курьер видит:

* номер заказа;
* адрес;
* ориентир;
* геолокацию;
* телефон для звонка;
* состав заказа, если это нужно для доставки;
* способ оплаты только в допустимом объеме;
* текущий статус доставки;
* инструкцию или заметку менеджера, если она есть.

Если менеджер написал дополнительную заметку по маршруту, курьер получает ее через канал связи с менеджером.

Курьер не видит:

* чужие заказы;
* клиентскую базу;
* пол клиента;
* желтый и красный список как внутренние данные;
* лишние платежные детали;
* внутренние записи отдела урегулирования.

---

## 8. Порядок объезда

Менеджеру не обязательно указывать порядок объезда клиентов.

Система может предложить маршрут, но курьер сам выбирает фактический порядок объезда, исходя из:

* дороги;
* расстояния;
* текущей ситуации на месте;
* доступности клиентов;
* сохранения качества еды;
* инструкций менеджера, если они были отправлены через канал связи.

На первом этапе порядок объезда остается решением курьера. Автоматический обязательный расчет маршрута не требуется.

---

## 9. Каждый заказ остается отдельным

Даже если несколько заказов объединены в один маршрут, каждый заказ остается отдельной операционной единицей.

Для каждого заказа отдельно фиксируется:

* принятие курьером;
* выезд;
* прибытие;
* QR-подтверждение доставки или fallback;
* оплата;
* проблема доставки;
* закрытие заказа;
* история действий.

Нельзя закрывать весь маршрут одним действием, если отдельные заказы еще не доставлены и не оплачены.

---

## 10. Недоступный клиент внутри маршрута

Если курьер в маршруте с несколькими заказами прибыл к клиенту и клиент недоступен, действует базовое правило доставки:

* курьер ждет 5 минут;
* после этого отмечает `Клиент недоступен`;
* менеджер получает красный код срочности;
* ситуация может уйти в отдел урегулирования.

Если у курьера есть следующие заказы, он не должен долго ждать одного клиента сверх базового ожидания. Менеджер должен дать инструкцию: продолжать маршрут, вернуться позже, вернуть заказ в кафе или ждать дополнительного решения.

Если следующих заказов нет и ситуация позволяет, курьер может ждать дольше по уже утвержденным правилам доставки.

---

## 11. Клиентское отображение

Клиент не видит, что его заказ объединен с другими заказами в один маршрут.

Клиент видит только свои этапы:

* `Передан курьеру`;
* `Уже в пути`;
* `Доставлен`;
* уведомления по своему заказу.

Для клиента не должно быть разницы, едет заказ напрямую или в попутном маршруте. Если из-за попутной доставки возникает задержка, клиенту не нужен отдельный технический статус. Менеджер может объяснить ситуацию через чат или телефон.

---

## 12. История действий

Система должна фиксировать:

* было ли объединение предложено системой;
* кто подтвердил объединение заказов в маршрут;
* какие заказы вошли в маршрут;
* когда маршрут был назначен курьеру;
* была ли отправлена заметка менеджера курьеру;
* когда курьер принял маршрут;
* действия курьера по каждому заказу;
* проблемные события по каждому заказу;
* когда каждый заказ был закрыт.

История маршрута не заменяет историю каждого отдельного заказа.

---

## 13. Требования к следующим документам

### 13.1 Technical Spec

Technical Spec должен раскрыть:

* как хранится группа заказов в одном маршруте;
* как ограничивается маршрут максимум тремя заказами;
* как система предлагает объединение заказов по локациям;
* как менеджер подтверждает или отклоняет предложение системы;
* как назначается несколько заказов одному курьеру;
* как ограничивается доступ курьера только к назначенным заказам;
* как система предлагает маршрут без обязательного порядка объезда;
* как работает канал связи менеджера с курьером;
* как фиксируются действия по каждому заказу отдельно;
* как маршрут отображается менеджеру;
* как маршрут отображается курьеру;
* как работает недоступный клиент внутри маршрута;
* как закрывается доступ курьера к данным после завершения каждого заказа;
* как не раскрывать клиенту внутреннюю группировку маршрута.

### 13.2 User Stories

User Stories должны проверить:

* система предлагает объединить заказы по локациям;
* менеджер подтверждает попутную доставку;
* менеджер отклоняет предложенное объединение;
* менеджер назначает одному курьеру два или три заказа;
* курьер сам выбирает очередность;
* менеджер отправляет курьеру заметку по маршруту;
* курьер видит только назначенные заказы;
* курьер доставляет первый заказ;
* курьер доставляет второй заказ;
* первый клиент недоступен, а у курьера есть следующий заказ;
* клиент не видит, что заказ едет в попутном маршруте;
* заказ из красного списка включается в маршрут после подтвержденной онлайн-оплаты;
* каждый заказ закрывается отдельно;
* данные клиента скрываются после завершения конкретной доставки.

---

## 14. Критерии готовности функции

Функция готова к переходу в User Stories и Technical Spec, если выполнены следующие условия:

* менеджер может объединить до трех готовых заказов в один маршрут;
* система может предложить объединение заказов по локациям;
* система не объединяет заказы автоматически без решения менеджера;
* курьер видит только назначенные ему заказы;
* каждый заказ в маршруте остается отдельной доставкой;
* QR, оплата, проблема и закрытие фиксируются отдельно по каждому заказу;
* курьер сам выбирает фактический порядок объезда;
* менеджер может отправить курьеру заметку через канал связи;
* клиент не видит внутреннюю группировку маршрута;
* попутная доставка доступна для желтого и красного списка, если выполнены условия оплаты и доставки;
* приоритеты и запреты по конкретным заказам остаются решением менеджера, без автоматического решения системы;
* при недоступном клиенте курьер не задерживает остальные заказы без инструкции менеджера;
* история маршрута и история каждого заказа сохраняются.

---

## 15. Открытые вопросы

Открытых вопросов по этой функции на текущем уровне нет.

Закрытые решения:

1. На первом этапе попутный маршрут может включать до трех заказов.
2. Система может предложить менеджеру объединение заказов по локациям, но финальное решение принимает менеджер.
3. Менеджер не обязан указывать порядок объезда. Система может предложить маршрут, а курьер сам решает, как ехать.
4. Менеджер может отправить курьеру дополнительную заметку через канал связи с курьером.
5. Попутная доставка доступна для заказов из желтого и красного списка, если выполнены условия оплаты и доставки.
6. Клиенту не раскрывается, что заказ едет в попутном маршруте.
7. Приоритеты и запреты по конкретным заказам остаются решением менеджера, без автоматического решения системы.
