# Карта спецификаций проекта

> **Методология**: сначала спецификации, потом планы, потом код.
> **Текущий статус**: Global Spec, Functional Map, основной набор Feature Specs, Technical Specs и User Stories зафиксированы; начата подготовка Work Plans.

---

## 1. Назначение

Эта папка хранит продуктовые документы проекта в единой структуре. Каждый следующий уровень должен опираться на предыдущий и не менять его смысл без отдельного утверждения.

Правила работы со спецификациями зафиксированы в `RULES.md`.

---

## 2. Текущие документы

1. `01_global_spec/Global_Spec.md` - глобальное видение проекта, бизнес-цели, роли и фундаментальные принципы.
2. `02_functional_map/Functional_Map.md` - карта возможностей, участников, пользовательских путей, ситуаций, MVP-границ и следующих шагов.
3. `03_feature_specs/` - спецификации отдельных функций продукта.
4. `04_technical_specs/` - технические спецификации реализации, доступов, данных, интеграций и безопасности.
5. `06_user_stories/` - пользовательские истории, сценарии и критерии приемки.
6. `07_work_plans/` - пошаговые планы реализации.

---

## 3. Планируемые уровни

Следующие разделы должны быть добавлены после подготовки первого Work Plan:

1. `05_visual_rules_skills/` - визуальные правила, интерфейсные соглашения и правила для AI-разработки.

---

## 4. Правила работы

* Если функции нет в спецификациях, ее нельзя переносить в план разработки.
* Если функция противоречит Global Spec, нужно остановиться и уточнить решение.
* Если Functional Map оставляет вопрос открытым, его нужно закрыть до Feature Spec или явно перенести в более поздний этап.
* Technical Specs не должны расширять бизнес-смысл функции, только предлагать реализацию.
* Work Plans должны ссылаться на утвержденные Feature Specs и User Stories.

---

3. После закрытия блокеров подготовить техническое решение перед реализацией MVP.
