## 1. Назначение

Этот документ описывает технические правила доступа к клиентским данным, маскирование, аудит раскрытия данных и правила совпадения признаков для желтого и красного списка.

Главный принцип: сотрудник видит только те данные, которые нужны его роли и конкретной ситуации. Доступ к клиентским данным должен быть контекстным, ограниченным и журналируемым.

---

## 2. Роли первого уровня

На первом этапе система использует следующие роли:

1. `Администратор`
2. `Менеджер`
3. `Курьер`
4. `Отдел урегулирования`
5. `Клиент`
6. `Система`

Отдельная роль `Старший менеджер` на первом этапе не нужна.

Администратор и владелец на этом уровне считаются одной управленческой ролью внутри платформы.

---

## 3. Базовая модель доступа

Доступ должен строиться по трем условиям:

1. **Роль** - что сотрудник вообще может делать.
2. **Контекст** - к какому заказу, жалобе, ситуации или товару относится действие.
3. **Состояние** - находится ли заказ, доставка, жалоба или ситуация в статусе, где доступ оправдан.

Если нет роли, контекста или состояния, доступ должен быть закрыт.

Пример:

* курьер может видеть телефон клиента только по назначенному ему заказу;
* после закрытия доставки курьер теряет доступ к телефону, адресу, ориентиру и геолокации;
* если курьеру снова нужны данные, он запрашивает их через менеджера.

---

## 4. Матрица доступа

| Данные / действие | Администратор | Менеджер | Курьер | Отдел урегулирования |
| --- | --- | --- | --- | --- |
| Просмотр текущих заказов | Да | Да | Только назначенные | Только переданные в урегулирование |
| Телефон клиента | Да, структурно | Да, по заказу | Только назначенная доставка | Да, по ситуации |
| Адрес и ориентир | Да, структурно | Да, по заказу | Только назначенная доставка | Да, по ситуации |
| Геолокация | Да, структурно | Да, по заказу | Только назначенная доставка | Да, по ситуации |
| Telegram / Email | Да, структурно | Ограниченно по заказу | Нет | Да, по ситуации |
| Пол / формат обращения | Не как обычное поле заказа | Нет | Нет | Да, для маршрутизации |
| Клиентский чат | Да, структурно | Да, по заказу | Нет | Только если передано в ситуацию |
| Жалобы | Да | Нет как управление | Нет | Только если администратор использует отдел как ресурс |
| Желтый список | Да, просмотр | Предупреждение по заказу | Нет | Полное управление |
| Красный список | Да, просмотр | Предупреждение по заказу | Нет | Полное управление |
| Управление меню | Да | Только доступность блюд | Нет | Нет |
| Скидочные товары | Да | Только при выданном праве | Нет | Нет |
| Закрытие жалоб | Да | Нет | Нет | Нет |
| Снятие желтого/красного списка | Нет | Нет | Нет | Да |

Администратор видит данные через структурированные разделы и журналы, но это не означает свободный неограниченный экспорт клиентской базы.

## 5. Маскирование данных

Маскирование нужно для обычного рабочего режима, где сотруднику не требуется видеть полные данные.

Минимальные правила:

* телефон показывается полностью только там, где нужен звонок или контакт;
* Telegram и Email не показываются курьеру;
* пол клиента не показывается менеджеру и курьеру;
* желтый и красный список не показываются курьеру;
* внутренние причины риска не показываются клиенту;
* платежные детали не показываются кухне и курьеру сверх нужного сообщения;
* кухня не видит телефон, адрес, геолокацию, Telegram, Email и оплату.

Примеры маскирования:

* телефон в списке заказов менеджера: `+20 *** *** 1234`, если полный номер не нужен прямо сейчас;
* email в журнале администратора: `a***@domain.com`, если полный email не нужен для действия;
* Telegram: показывать как подтвержденный канал без раскрытия лишних деталей, если не требуется контакт.

Точные форматы маскирования могут быть уточнены в UI/User Stories, но принцип должен сохраняться: полные данные раскрываются только при необходимости.

---

## 6. Раскрытие данных

Раскрытие данных - это действие, при котором сотрудник получает полный контакт, адрес, геолокацию или другой чувствительный элемент, который обычно скрыт.

Раскрытие должно быть:

* привязано к роли;
* привязано к заказу, доставке, жалобе или ситуации;
* ограничено временем и состоянием;
* записано в журнал.

Минимальные события раскрытия:

* менеджер открывает полный телефон клиента;
* менеджер открывает полную геолокацию;
* курьер получает данные назначенной доставки;
* курьер теряет данные после закрытия доставки;
* отдел урегулирования открывает Telegram/Email/телефон клиента;
* администратор открывает структурированные клиентские данные;
* менеджер передает ситуацию в урегулирование;
* сотрудник запрашивает повторное раскрытие данных после закрытия доставки.

---

## 7. Аудит

Система должна вести журнал действий по доступу и важным операциям.

Минимальные поля журнала:

* дата и время;
* сотрудник или системный actor;
* роль;
* действие;
* объект действия;
* причина или контекст;
* результат;
* старое значение и новое значение, если было изменение;
* источник действия: интерфейс, системное событие, автоматический процесс.

Журнал должен фиксировать:

* вход сотрудника в систему;
* создание, изменение и отключение сотрудника;
* назначение роли;
* раскрытие клиентских данных;
* назначение курьера;
* передачу заказа на кухню;
* подтверждение оплаты;
* QR/fallback подтверждение доставки;
* передачу в урегулирование;
* действия с желтым и красным списком;
* действия с жалобами;
* изменение цены в рабочее время через усложненный сценарий;
* выдачу менеджеру права на скидочный товар.

---

## 8. Курьерский доступ

Курьер получает данные только по назначенным доставкам.

Курьер может видеть:

* номер назначенного заказа;
* адрес;
* ориентир;
* геолокацию;
* телефон для звонка;
* состав заказа, если это нужно для доставки;
* итоговую сумму только при наличной оплате;
* сообщение `Оплату брать не нужно`, если электронная оплата подтверждена;
* инструкцию менеджера, если она есть.

Курьер не может видеть:

* клиентскую базу;
* чужие заказы;
* Telegram;
* Email;
* пол клиента;
* внутренние причины желтого/красного списка;
* историю клиента;
* жалобы;
* платежные детали;
* данные после закрытия доставки.

После закрытия доставки система должна закрыть доступ курьера к телефону, адресу, ориентиру и геолокации по этому заказу.

---

## 9. Менеджерский доступ

Менеджер работает с операционными заказами.

Менеджер видит:

* подтвержденные рабочие заказы;
* данные, нужные для обработки заказа;
* адрес, ориентир и геолокацию;
* телефон для связи и доставки;
* клиентский чат по заказу;
* способ оплаты и состояние оплаты;
* предупреждение о риске, если заказ похож на желтый/красный случай;
* вкладки `На доставке` и `На урегулировании`;
* счетчики непрочитанных сообщений.

Менеджер не видит:

* неподтвержденные попытки как обычные заказы;
* пол клиента как обычное поле заказа;
* полную внутреннюю логику желтого/красного списка;
* жалобы как управляемые задачи;
* клиентскую базу вне рабочего контекста.

Если менеджер передает ситуацию в отдел урегулирования, система сама маршрутизирует ситуацию в нужное направление. Менеджер не выбирает мужской или женский отдел вручную.

---

## 10. Доступ отдела урегулирования

Отдел урегулирования получает доступ только в рамках конкретной ситуации.

Отдел урегулирования видит:

* неподтвержденные попытки;
* ситуации, переданные менеджером;
* ситуации, возникшие при недоступном клиенте;
* Telegram/Email/телефон/адрес/геолокацию по ситуации;
* дополнительные контакты, если клиент указал их добровольно;
* признаки желтого/красного списка;
* историю связанных проблем.

Отдел урегулирования может:

* связаться с клиентом;
* вернуть ситуацию менеджеру;
* архивировать неподтвержденную попытку;
* снять запись с желтого списка;
* снять запись с красного списка;
* зафиксировать результат контакта;
* принять финальное решение о возврате заказа.

---

## 11. Администраторский доступ

Администратор имеет широкие права внутри платформы, но не имеет права заходить в техническую часть, менять код или выполнять действия вне продуктовой платформы.

Администратор может:

* управлять сотрудниками;
* назначать роли;
* отключать и восстанавливать доступ;
* управлять меню и ценами;
* управлять скидочными товарами;
* выдавать менеджеру право на конкретный скидочный товар;
* видеть жалобы;
* закрывать, архивировать, перемещать и удалять жалобы;
* видеть структурированные клиентские данные;
* видеть журналы действий.

Администратор не должен:

* снимать желтый или красный список вместо отдела урегулирования;
* превращать ограниченные роли в свободный доступ к клиентской базе;
* обходить утвержденный процесс жалоб;
* менять технический код системы из платформы.

---

## 12. Желтый и красный список

Желтый и красный список строятся не по одному признаку, а по набору совпадений.

Базовые признаки для сравнения:

* геолокация;
* адрес;
* телефон;
* Telegram;
* Email;
* клиентский профиль;
* устройство;
* имя;
* история заказов;
* заметки или дополнительные контакты.

Ориентир первого уровня: риск считается похожим, если совпадает несколько признаков из общего набора. Для желтого списка ориентир - 3-4 совпадающих признака, но финальный технический порог должен быть настраиваемым.

Красный список возникает после повторения проблемного случая или по решению отдела урегулирования в рамках утвержденных причин:

* клиент недоступен;
* конфликт;
* ложный заказ;
* отказ от оплаты.

---

## 13. Правило совпадения признаков

Технически совпадение признаков должно считаться не как точное равенство одной строки, а как набор нормализованных сравнений.

Примерная логика:

1. Система собирает признаки нового заказа.
2. Система нормализует признаки: телефон, адрес, email, Telegram, геолокацию.
3. Система сравнивает новый заказ с активными риск-записями.
4. Каждое совпадение дает вес.
5. Если общий вес превышает порог, менеджер видит предупреждение.
6. Если риск высокий, ситуация может быть отправлена в отдел урегулирования.

На первом этапе система не должна автоматически отправлять подтвержденный рабочий заказ в красный список только по совпадениям. Она должна предупреждать менеджера и фиксировать риск. Решение по спискам остается у отдела урегулирования.

---

## 14. Хранение и сроки

Минимальные сроки из Feature Specs:

* доверенный Telegram/Email действует 1 год после успешной верификации;
* неподтвержденная попытка уходит в архив через 24 часа, если не решена;
* желтый список хранится 60 дней;
* красный список архивируется через 24 часа, но риск остается доступным системе;
* жалобы хранятся 1 год;
* чат заказа хранится 30 дней после закрытия заказа.

Техническая реализация должна учитывать эти сроки как правила жизненного цикла данных.

---

## 15. Требования к интерфейсам

Интерфейсы должны поддерживать:

* показ замаскированных данных по умолчанию;
* раскрытие данных только через действие;
* явное отображение предупреждений риска менеджеру;
* закрытие доступа курьера после завершения доставки;
* структурированные разделы данных для администратора;
* отдельную вкладку жалоб для администратора;
* отдельный контур урегулирования;
* журнал действий.

Точные UI-компоненты раскрываются позже в Visual Rules, User Stories или Work Plans.

---

## 16. Критерии готовности

Technical Spec готов к переходу в User Stories и Work Plans, если:

* роли первого уровня определены;
* доступ строится по роли, контексту и состоянию;
* матрица доступа покрывает основные данные;
* курьер не получает клиентскую базу;
* менеджер не видит пол клиента как обычное поле заказа;
* отдел урегулирования получает данные только по ситуации;
* администратор работает внутри платформы, но не получает техническую закулису;
* раскрытие клиентских данных журналируется;
* желтый/красный список строятся по набору признаков;
* сроки хранения данных закреплены.

---

## 17. Закрытые решения

1. Полный номер телефона показывается через ручное действие `Показать полный номер`, если полный номер не нужен роли автоматически. В карточке активной доставки курьера полный номер может быть показан сразу, потому что звонок является прямой задачей доставки.
2. Экспорт клиентских данных на первом этапе запрещен для всех ролей, включая администратора. Если позже экспорт понадобится, он должен быть отдельной функцией с ограничением прав, причиной экспорта и журналом.
3. Журналируется не каждое открытие карточки, а раскрытие чувствительных данных и важные действия. Это снижает шум журнала и сохраняет контроль над реальным риском.
4. На первом этапе используется правило `3-4 совпадения` без сложных весов. При этом техническая структура должна позволять добавить веса признаков позже.
5. Журнал раскрытия клиентских данных хранится 1 год. Для событий высокого риска, связанных с желтым/красным списком, жалобами и урегулированием, запись может храниться вместе с соответствующим кейсом по его срокам хранения.
