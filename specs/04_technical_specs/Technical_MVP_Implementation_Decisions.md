> **Статус**: Зафиксировано для MVP перед кодом

## 1. Назначение

Документ закрывает технические блокеры перед началом кода для первого MVP-потока заказа.

Решения ниже не добавляют новые бизнес-функции. Они выбирают минимальный технический способ реализовать уже утвержденные спецификации: витрину, корзину, оформление заказа, CRM менеджера, кухню, курьера, доставку, оплату, клиентскую страницу, чат, урегулирование, скидочную продажу и жалобы.

---

## 2. Принцип первого MVP

Первый MVP должен быть построен как один продуктовый контур, а не как набор разрозненных сервисов.

Приоритеты:

1. быстрый переход к рабочему прототипу;
2. единая модель данных;
3. строгие роли и контекстный доступ;
4. возможность развивать систему без переписывания доменной логики;
5. минимум внешних интеграций в первом релизе.

---

## 3. Решение по стеку

Для первого MVP выбирается:

1. **Frontend и backend**: Next.js с TypeScript.
2. **UI**: React-компоненты внутри Next.js.
3. **API**: серверные обработчики Next.js для внутренних действий приложения.
4. **База данных**: PostgreSQL.
5. **Доступ к данным**: Prisma.
6. **Фоновые задачи первого уровня**: серверные задачи приложения или отдельный простой worker внутри той же кодовой базы.
7. **Валидация данных**: единые TypeScript-схемы на границе форм и API.

Причина выбора Next.js:

* проекту нужны публичные страницы клиента, внутренние CRM-экраны, курьерский кабинет, клиентская страница заказа и серверная логика в одном MVP;
* один стек снижает количество инфраструктурных решений перед первым релизом;
* TypeScript позволяет держать общие типы для заказа, статусов, ролей и событий;
* Next.js подходит для постепенного роста: сначала монолит, позже отдельные сервисы можно выделять по необходимости.

Angular на первом этапе не выбирается, потому что для него нужен отдельный backend-контур. Это оправдано для крупной enterprise-панели, но для текущего MVP увеличит стартовую сложность без прямой пользы.

---

## 4. Решение по хранению данных

Для MVP используется PostgreSQL как основное хранилище.

Минимальные правила:

1. все рабочие сущности заказа хранятся в PostgreSQL;
2. `OrderStatusHistory` и `OrderEventLog` являются append-only;
3. чувствительные данные хранятся отдельно от обычных рабочих данных;
4. для поиска доверенных каналов используются нормализованные значения и хэши;
5. открытые Telegram, Email и телефон не используются как публичные идентификаторы;
6. удаление важных рабочих записей заменяется архивированием, если спецификация не требует другого поведения.

Redis, очередь сообщений и отдельное файловое хранилище на первом этапе не являются обязательными. Их можно добавить позже, если появится реальная нагрузка или необходимость в надежных фоновых очередях.

## 5. Решение по OTP

OTP реализуется через два канала:

1. Telegram Bot API для Telegram;
2. SMTP или транзакционный Email-провайдер для Email.

Минимальная логика:

1. сервер создает `VerificationRequest`;
2. сервер генерирует одноразовый код;
3. в базе хранится только хэш OTP;
4. код действует 10 минут;
5. повторная отправка доступна через 60 секунд;
6. после 3 неправильных попыток включается блокировка на 15 минут;
7. после успешной проверки создается или обновляется `TrustedChannel`;
8. доверенный канал действует 1 год.

API-обработчик отправки OTP должен использовать rate limiting middleware до генерации кода и до внешней отправки. Минимальные ограничения: по IP, по устройству/сессии и по выбранному Telegram/Email-каналу. Если лимит превышен, OTP не создается, Telegram Bot API или Email-провайдер не вызывается, а событие фиксируется как защитное ограничение.

Для локальной разработки разрешен dev-режим: OTP не отправляется наружу, а показывается только в серверном логе разработки. В production такой режим должен быть отключен.

## 6. Решение по QR доставки

QR подтверждает доставку, но не оплату.

Технический способ:

1. при назначении курьера система создает одноразовый `DeliveryConfirmationToken`;
2. открытое значение токена показывается только внутри QR;
3. в базе хранится хэш токена, `orderId`, `courierId`, `deliveryAssignmentId`, срок жизни и статус;
4. QR ведет на серверный route подтверждения доставки;
5. после успешного сканирования токен становится недействительным;
6. после закрытия заказа токен также становится недействительным;
7. fallback требует обязательную причину и отдельное событие журнала.

QR нельзя использовать повторно и нельзя использовать для другого заказа или другого назначения курьера.

---

## 7. Решение по клиентской странице заказа

Клиентская страница заказа доступна без полноценной регистрации.

Технический способ:

1. после создания `Order` система создает `ClientOrderAccess`;
2. клиент получает локальный безопасный токен устройства;
3. токен хранится на стороне клиента как локальный доступ к конкретному заказу;
4. на сервере хранится только хэш токена и связь с заказом;
5. токен не содержит персональные данные;
6. доступ действует для активного заказа и 30 дней после закрытия;
7. при открытии с другого устройства клиент вводит номер заказа и проходит дополнительную проверку подтвержденного канала.

Номер заказа сам по себе не является достаточным секретом для доступа к чату и деталям заказа.

---

## 8. Решение по ролям и правам

Для MVP используется ролевая модель из `Technical_Access_Audit.md`:

1. `Администратор`;
2. `Менеджер`;
3. `Курьер`;
4. `Отдел урегулирования`;
5. `Клиент`;
6. `Система`.

Проверка доступа строится не только по роли, но и по контексту:

1. роль пользователя;
2. связанный объект: заказ, доставка, жалоба или ситуация;
3. текущее состояние объекта;
4. назначение сотрудника на конкретную задачу;
5. необходимость раскрытия чувствительных данных.

На первом этапе не создается роль `Старший менеджер`. Если позже она понадобится, ее нужно добавить отдельной спецификацией.

---

## 9. Решение по журналированию

Для MVP используется единый append-only журнал важных действий.

Минимальная сущность: `AuditLog`.

Минимальные поля:

* `id`;
* `actorId`;
* `actorRole`;
* `action`;
* `objectType`;
* `objectId`;
* `context`;
* `result`;
* `oldValue`;
* `newValue`;
* `createdAt`;
* `source`.

Журналируются:

1. вход сотрудника;
2. изменение ролей и доступов;
3. раскрытие чувствительных данных;
4. создание и подтверждение заказа;
5. передача на кухню;
6. повторная печать кухонного чека;
7. назначение курьера;
8. QR или fallback подтверждение доставки;
9. подтверждение оплаты;
10. отправка платежной ссылки;
11. передача в урегулирование;
12. действия с желтым и красным списком;
13. действия с жалобами;
14. выдача права на скидочный товар.

Обычное открытие карточки без раскрытия чувствительных данных не журналируется как отдельное событие.

---

## 10. Решение по ручной платежной ссылке

Автоматическая платежная интеграция не входит в первый MVP.

Технический способ:

1. менеджер создает платежную ссылку во внешнем платежном инструменте;
2. менеджер вручную отправляет ссылку клиенту через утвержденный рабочий процесс;
3. система фиксирует факт отправки ссылки;
4. открытая ссылка не хранится дольше операционной необходимости;
5. в истории хранится безопасный идентификатор или маска ссылки;
6. срок жизни ссылки для первого MVP - 30 минут;
7. оплату подтверждает менеджер вручную;
8. при подтверждении менеджер вводит короткий идентификатор платежа или служебную заметку.

Курьер не подтверждает электронную оплату. Курьер только видит, нужно ли брать наличные.

## 11. Архитектурные границы MVP

Первый релиз строится как модульный монолит.

Минимальные доменные модули:

1. `menu`;
2. `cart`;
3. `orders`;
4. `verification`;
5. `crm`;
6. `kitchen`;
7. `delivery`;
8. `payments`;
9. `client-page`;
10. `chat`;
11. `resolution`;
12. `complaints`;
13. `audit`;
14. `users-and-access`.

Модули могут жить в одной кодовой базе, но не должны напрямую обходить правила доступа и журналирования.

---

## 12. Что не входит в первый технический шаг

В первый технический шаг не входит:

1. полноценная автоматическая платежная интеграция;
2. отдельный backend-сервис вне Next.js;
3. микросервисная архитектура;
4. мобильное приложение;
5. push-уведомления;
6. сложная аналитика;
7. экспорт клиентской базы;
8. финальный UI-kit;
9. автоматический выбор маршрута курьера;
10. автоматическое управление складом.

## 14. Критерии готовности

Документ готов к использованию перед кодом, если:

* решения не противоречат существующим Feature Specs, Technical Specs и User Stories;
* каждый блокер Work Plan имеет конкретное техническое решение;
* выбранный стек позволяет реализовать публичные и внутренние страницы в одном MVP;
* модель данных остается совместимой с `Technical_Order_Data_Model.md`;
* OTP, QR, платежная ссылка и клиентский доступ не раскрывают персональные данные в открытом виде;
* роли и журналирование соответствуют `Technical_Access_Audit.md`.
