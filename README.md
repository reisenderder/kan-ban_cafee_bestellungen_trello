# kan-ban_cafee_bestellungen_trello

## Универсальные правила проекта

Эти правила действуют для всего проекта и проверяются перед каждым следующим шагом.

1. Документация, спецификации, планы и пользовательские тексты пишутся на русском языке.
2. Проект ведется по алгоритму: `Spec -> Plan -> Code`.
3. Нельзя реализовывать функцию, если она не описана в утвержденной спецификации.
4. Если новая идея противоречит Global Spec, работа останавливается до уточнения.
5. Корневые правила важнее правил папок; правила папок уточняют порядок работы внутри своего раздела.
6. Клиентские данные должны раскрываться только по роли, задаче и текущему контексту.
7. Любое новое правило добавляется только если оно предотвращает ошибку, экономит время или делает процесс стабильнее.
8. После сообщения `pull request смержен` работа синхронизируется с актуальной веткой `main`; каждая новая задача начинается в новой ветке от `main`.

## Структура

* `.agents/` - правила и материалы для AI-агентов.
* `specs/` - спецификации, функциональные карты и продуктовая документация.
* `work_plans/` - планы разработки и история их выполнения.
* `work_plans/active/` - активные планы, работа по которым еще не завершена или еще не подтверждена пользовательским тестированием.
* `work_plans/completed/` - завершенные планы, сохраненные как история выполненных работ.
* `roadmap/` - хронологическая история фактически завершенных работ по областям проекта.

## Жизненный цикл планов разработки

1. Новый план разработки создается в `work_plans/active/`.
2. Во время реализации план остается в `active/`, даже если код уже написан или автоматические проверки проходят.
3. План считается завершенным только после того, как пользователь:
   * полностью протестировал результат;
   * подтвердил, что реализация работает;
   * попросил подготовить публикацию изменений через commit, push или pull request.
4. Перед выполнением `git add`, `commit`, `push` или созданием pull request соответствующий файл плана обязательно перемещается из `active/` в `completed/`.
5. Одновременно в соответствующие журналы `roadmap/` добавляются хронологические записи о фактически выполненных работах:
   * данные;
   * frontend и дизайн;
   * искусственный интеллект и AI-анализ.
6. Если план затрагивает несколько областей, запись добавляется в каждый соответствующий журнал.
7. При перемещении обновляются статус плана, индекс `work_plans/README.md` и все ссылки на файл.
8. Завершенный план и записи roadmap не удаляются: они сохраняют историю разработки.

AI-агент не переносит план в `completed/` только на основании завершения кода или автоматических тестов. Для переноса требуется явное подтверждение пользователя после ручного тестирования.

Обновление roadmap является обязательной частью завершения плана и выполняется до Git-публикации изменений. В roadmap записываются только выполненные и проверенные работы, а не будущие намерения.
