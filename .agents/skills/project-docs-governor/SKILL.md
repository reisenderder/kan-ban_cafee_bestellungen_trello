# Project Docs Governor

Используй этот skill, чтобы держать документацию проекта в правильной иерархии и не допускать, чтобы код, планы или новые решения уходили в сторону от утвержденных спецификаций.

## First Checks

1. Проверь текущую Git-ветку и рабочее дерево перед изменением файлов.
2. Определи, относятся ли существующие изменения к текущей задаче или к предыдущему шагу.
3. Перед правками документации прочитай `.agents/README.md`, `specs/RULES.md` и `specs/README.md`.
4. После верхних правил читай только релевантные исходные документы.
5. Перед переходом от анализа к правкам, от правок к проверке и от проверки к Git-действиям коротко сообщай пользователю, что будет сделано дальше.

## Source Order

Соблюдай иерархию документации:

1. Global Spec: цель продукта, роли, базовые принципы.
2. Functional Map: зоны продукта, потоки, границы, открытые области.
3. Feature Specs: утвержденное поведение функций.
4. Technical Specs: техническая реализация без изменения бизнес-смысла.
5. User Stories: сценарии пользователей и критерии приемки.
6. Work Plans: порядок реализации, зависимости и критерии готовности.
7. Code: только после того, как документационный слой разрешает изменение.

Не позволяй нижнему уровню менять смысл верхнего уровня. Если нижний документ конфликтует с верхним, остановись и сообщи о конфликте.

## Task Routing

Используй эти правила маршрутизации:

1. Новое продуктовое поведение сначала относится к Feature Specs, если оно еще не утверждено.
2. Технический механизм или архитектурное решение относится к Technical Specs.
3. Сценарий, критерии приемки и edge cases относятся к User Stories.
4. Порядок реализации, зависимости и readiness gates относятся к Work Plans.
5. Навигация между документами относится к README-файлам.
6. Правила работы AI-агентов относятся к `.agents`.
7. Долгосрочные инженерные правила всего репозитория относятся к `AGENTS.md`, если такой файл будет добавлен позже.

Для подробных алгоритмов читай `references/documentation-workflow.md`.

## Required Reports

При работе с документацией коротко сообщай:

1. что проверено;
2. что уже было покрыто;
3. какой пробел остался;
4. какие файлы должны измениться;
5. что изменилось после правок;
6. какая проверка выполнена;
7. выполнялись ли Git-действия.

Не выполняй `git add`, `commit` или `push`, пока пользователь явно не попросит соответствующий этап.

## Validation

После правок проверь:

1. нет случайных `TODO`, `TBD` или `FIXME`, если пользователь явно не просил оставить открытый маркер;
2. затронутые README-индексы корректно отражают текущие документы;
3. измененный слой не добавляет поведение, не поддержанное верхними слоями;
4. следующий шаг понятен;
5. Git status содержит только ожидаемые файлы.

Для проверок согласованности читай `references/consistency-checks.md`.

## Language And Style

Пиши проектную документацию на русском языке, кроме имен файлов, code identifiers, branch names, названий frameworks и API terms.

Держи документы практичными и прямыми. Предпочитай конкретные решения, границы, критерии и следующие действия вместо общих рассуждений.
