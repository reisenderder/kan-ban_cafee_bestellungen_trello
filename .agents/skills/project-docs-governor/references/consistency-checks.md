# Consistency Checks

## Purpose

Используй этот reference для аудита документации, анализа пробелов и задач вида "это уже покрыто?".

## Basic Check

1. Ищи точные термины и синонимы через `rg`.
2. Читай найденные документы с достаточным контекстом.
3. Разделяй:
   * уже покрыто;
   * частично покрыто;
   * отсутствует;
   * противоречит;
   * дублируется, но не мешает.
4. Меняй только самый узкий набор файлов, нужный для закрытия пробела.
5. Обновляй README-индексы только если изменился список документов или текущий статус.

## Required Search Patterns

Перед правками используй точечный поиск:

* status/routing: `status|статус|workflow|этап|переход`;
* access/security: `access|доступ|роль|аудит|журнал|маскир`;
* verification: `OTP|Telegram|Email|вериф|подтвержд|лимит|rate|IP|сесс|устрой`;
* payment: `оплат|платеж|payment|link|ссылка|налич`;
* delivery: `курьер|достав|QR|fallback|геолокац`;
* client page/chat: `чат|client|страниц|tracking|Прочитано`;
* unresolved markers: `TODO|TBD|FIXME`;
* открытые вопросы: `открыт|не решен|уточнить|блокер`.

## Conflict Rules

Если два документа противоречат друг другу:

1. Global Spec имеет приоритет над Functional Map.
2. Functional Map имеет приоритет над Feature Specs.
3. Feature Specs имеют приоритет над Technical Specs.
4. Technical Specs ограничивают User Stories и Work Plans.
5. Work Plans не создают новое поведение.

Не решай бизнес-конфликт молча через правку нижнего документа. Сообщи о конфликте и запроси подтверждение, если верхний источник не дает однозначного решения.

## Done Criteria

Документационное изменение готово к пользовательскому просмотру, когда:

1. пробел закрыт в правильном слое;
2. не добавлено новое неподдержанное поведение;
3. затронутые индексы точны;
4. не осталось случайных unresolved markers;
5. Git status показывает только ожидаемые файлы;
6. пользователь получил короткий отчет перед Git-действиями.
